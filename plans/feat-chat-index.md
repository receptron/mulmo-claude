# Chat History Index & Search

## Goal

Let the user reference past conversations naturally — "あの前話したXXって何だっけ?" — instead of having to know exactly which session held the answer. The system maintains a small per-session index (title + summary + keywords) generated by Claude itself, refreshed periodically in the background, and exposed to the agent via a new MCP tool so Claude can grep across history when the user asks.

The same generated **title** also replaces the current "first user message" preview in the history pane, so the sidebar becomes more informative.

## User stories

- **As a user**, I open the history pane and each session shows a meaningful title ("ビリーズブートキャンプの 2 週間スケジュール") instead of the truncated first message ("2週間分の予定を入れて欲しいんだけど")
- **As a user**, I ask Claude "この前話してた wiki の SCHEMA の話、なんだっけ?" and Claude finds the relevant past session, summarizes its contents, and offers to load it
- **As an operator**, I never need to invoke or wait for indexing manually — it happens on server startup and every few hours in the background, and is incremental (only changed sessions are re-indexed)

## Architecture overview

```text
┌─ server start ────────────────────────┐
│ 1. initIndex() — load existing index  │
│ 2. scanAndQueue() — enqueue stale     │
│ 3. setInterval(scanAndQueue, 6h)      │
└─────────┬─────────────────────────────┘
          │
          ▼
┌─ Indexer worker (single-flight) ───────┐
│ for each queued session:                │
│   read chat/<id>.jsonl                  │
│   summarizeViaClaudeCli()               │
│   write chat/index/<id>.json           │
│   update chat/index/manifest.json      │
└─────────────────────────────────────────┘
                  ▲
                  │
                  │ "scan needed"
                  │
┌─ /api/agent finishes ──────────────────┐
│   indexer.markStale(chatSessionId)      │
└─────────────────────────────────────────┘

┌─ MCP tool: searchChatHistory ──────────┐
│ in: query (string), limit?: number     │
│ out: [{ id, title, summary, keywords,   │
│        startedAt, score, snippet }]    │
│  reads chat/index/manifest.json       │
└─────────────────────────────────────────┘
```

## Data model

### `chat/index/<id>.json` — one file per session

```ts
interface ChatIndexEntry {
  id: string;                    // session id
  roleId: string;                // copied from session meta
  startedAt: string;             // ISO
  // Hash of the source jsonl content at the time of indexing.
  // Used to detect stale entries.
  sourceSha256: string;
  sourceLines: number;           // line count when indexed
  indexedAt: string;             // ISO

  // Summary fields generated by claude
  title: string;                 // ~60 char user-facing title
  summary: string;               // ~200 char one-paragraph synopsis
  keywords: string[];            // 5–10 single-word topic tags
}
```

### `chat/index/manifest.json` — single file for fast lookup

```ts
interface ChatIndexManifest {
  version: 1;
  entries: ChatIndexEntry[];     // sorted by startedAt desc
}
```

The manifest is **incrementally updated** rather than rebuilt from scratch. When a session is indexed or re-indexed, only that one entry is upserted; the rest of the file is preserved. This keeps each indexer cycle cheap and bounds I/O for large indexes.

The MCP tool reads only the manifest, so search is one fs read + in-memory filter.

> The manifest is derivable from the per-session `index/<id>.json` files (rebuilt from scratch on corruption), but keeping it as a single live file makes search cheap (one read, no glob).

### Why a separate `chat/index/` directory

`chat/` already contains `.json` (per-session metadata) and `.jsonl` (per-session conversation) files. Putting auto-generated index files there would muddle file types. `chat/index/` keeps them grouped under their own subdir while still living next to the data they describe. The directory is **not** hidden — it's normal workspace content the user can browse.

### Storage scaling

- **Per-session files** are inherently sharded — adding a session adds one ~500-byte file. No upper limit other than the file system.
- **Manifest** grows linearly with session count. At 1000 sessions and ~500 bytes each it is ~500 KB, which still loads in milliseconds. Beyond ~10000 sessions we may want to shard by year (`manifest-2026.json`, `manifest-2027.json`) or drop the manifest in favour of an on-startup glob into memory. Out of scope for the first cut, called out as a Phase 4 follow-up.
- **No external database** — all storage is plain JSON on disk so it's grep-able and inspectable.

### 2. `server/chat-index/summarizer.ts` — claude CLI wrapper

```ts
interface SummaryResult {
  title: string;
  summary: string;
  keywords: string[];
}

export async function summarizeJsonl(
  jsonlPath: string,
  opts?: { timeoutMs?: number },
): Promise<SummaryResult>;
```

Implementation: spawn `claude --output-format json --system-prompt <system> -p <user>` (or use stream-json + parse the final assistant message). The system prompt instructs strict JSON output:

```text
You are a chat-summarization assistant. The user will paste the contents of one chat session. Read it and reply with a JSON object on a single line, no markdown, no explanation:

{ "title": "...", "summary": "...", "keywords": ["...","..."] }

Rules:
- title: at most 60 characters, in the same language as the conversation, capturing the main topic
- summary: at most 200 characters, one sentence, also same language
- keywords: 5 to 10 lowercase single-word or short-phrase tags useful for search
- Strict JSON only. No code fences. No trailing text.
```

The user prompt is the trimmed conversation text (see "Truncation" below).

### Truncation strategy

JSONL files can grow long. To cap CLI input + cost:

- Strip tool-result entries (keep only `source: "user"` and `source: "assistant"` text entries)
- Trim each message to ~500 chars
- Cap total input to ~8000 chars: take the first ~3000 + last ~5000 if longer (gives Claude both context and current state)

### 3. `server/chat-index/scheduler.ts` — periodic trigger

Configurable from environment variables so the user can tune refresh cadence and batch size without editing code:

| Var | Default | Meaning |
|---|---|---|
| `CHAT_INDEX_REFRESH_HOURS` | `6` | Hours between background refresh cycles. Set to `0` to disable periodic refresh (startup pass still runs). |
| `CHAT_INDEX_BATCH_SIZE` | `20` | Maximum sessions processed per cycle. Distributes work across multiple cycles when there is a large backlog. |
| `CHAT_INDEX_CONCURRENCY` | `1` | Number of `claude` CLI invocations in flight at once within a cycle. Default 1 = strictly sequential. |

```ts
import { indexStale } from "./indexer.js";

const refreshHours = Number(process.env.CHAT_INDEX_REFRESH_HOURS ?? 6);
const batchSize = Number(process.env.CHAT_INDEX_BATCH_SIZE ?? 20);
const concurrency = Number(process.env.CHAT_INDEX_CONCURRENCY ?? 1);

let timer: NodeJS.Timeout | null = null;
let running = false;

export async function startChatIndexer(): Promise<void> {
  // Initial pass on startup, async so it doesn't block server boot.
  void runOnce();
  if (refreshHours > 0) {
    timer = setInterval(runOnce, refreshHours * 60 * 60 * 1000);
  }
}

export function stopChatIndexer(): void {
  if (timer) clearInterval(timer);
}

async function runOnce(): Promise<void> {
  if (running) return; // single-flight
  running = true;
  try {
    const n = await indexStale({ limit: batchSize, concurrency });
    if (n > 0) console.log(`[chat-index] refreshed ${n} session(s)`);
  } catch (err) {
    console.error("[chat-index] refresh failed:", err);
  } finally {
    running = false;
  }
}
```

Wired from `server/index.ts` after `initWorkspace()`. Stops on SIGTERM.

**Distribution model**: a workspace with hundreds of un-indexed sessions does not get summarised in one shot. Each cycle picks `batchSize` stale sessions and processes them. Per-session files are written immediately as each summary completes, so an interrupted cycle does not lose work — the next cycle picks up where it left off. Combined with `refreshHours`, this naturally spreads load across the day.

The startup pass uses the same `batchSize` cap. To force a faster initial backfill, the operator can lower `CHAT_INDEX_REFRESH_HOURS` temporarily, or run multiple `runOnce()` cycles back-to-back via a future admin endpoint (not in scope here).

### 4. MCP tool: `searchChatHistory`

New plugin in `src/plugins/searchChatHistory/` (mirrors the existing local-plugin pattern from CLAUDE.md):

- `definition.ts` — `toolDefinition` only, no Vue
- `index.ts` — `execute()` calls `POST /api/chat-history/search`
- View / Preview — minimal: show a list of matching sessions with title + snippet, click to load that session

Server route:

```ts
// server/routes/chat-history.ts
router.post("/chat-history/search", (req, res) => {
  const { query, limit = 10 } = req.body;
  const manifest = readManifest();
  const scored = scoreEntries(manifest.entries, String(query));
  res.json({ results: scored.slice(0, limit) });
});
```

`scoreEntries` is plain in-memory text matching:

- Lowercase normalization
- Score = 5 × keyword exact match + 3 × title substring + 1 × summary substring
- Tie-break: more recent first
- Returns at most `limit` entries with `{ id, title, summary, keywords, startedAt, score, snippet }`

For the first cut **no semantic / vector search** — keyword match is plenty for "the wiki SCHEMA conversation" type queries and keeps the runtime dependency-free.

## Triggering & throttling

| Trigger | What it does |
|---|---|
| Server start | Kicks off `runOnce()` async (does not block start). Indexes up to `CHAT_INDEX_BATCH_SIZE` sessions whose source hash differs from their indexed hash. |
| `setInterval` every `CHAT_INDEX_REFRESH_HOURS` | Same as startup. Single-flight via the `running` boolean. |
| `searchChatHistory` MCP call | Reads manifest only; never triggers indexing. |

**No per-run trigger**: post-agent-run reindexing is intentionally NOT included. Freshly-finished sessions are picked up by the next periodic cycle. This keeps the indexer simple, avoids contention with active runs, and matches the user's goal of background-only processing.

**Cost guard**: each cycle is bounded by `CHAT_INDEX_BATCH_SIZE` (default 20). A backlog of hundreds of sessions is processed across many cycles instead of a single burst. Each session = one `claude` CLI invocation = one short chat-completion. With Haiku-class pricing, indexing 100 sessions costs in the low cents.

**Distributed processing**: per-session work is independent and per-session files are written immediately. An interrupted cycle, server restart, or crash mid-cycle never corrupts global state — the next cycle simply picks up the still-stale entries.

## Error handling

- `summarizeJsonl` failure (CLI error, JSON parse failure, timeout) → log to console, leave entry stale, retry next cycle
- `manifest.json` corruption → discard, rebuild from per-session `.index/<id>.json`
- Per-session `.index/<id>.json` corruption → re-summarize on next cycle
- `claude` CLI not available (e.g. user logged out) → log warning once per cycle, skip indexing entirely (`isClaudeAvailable()` check)

## Confirmed decisions

All design questions resolved with the user before implementation:

1. **Summarization mechanism**: spawn `claude` CLI (no new SDK dep, consistent with existing agent path).
2. **Refresh cadence**: default 6 hours, **configurable** via `CHAT_INDEX_REFRESH_HOURS` env var. `0` disables periodic refresh, leaving only the startup pass.
3. **Trigger model**: startup + periodic only. **No** post-run reindex hook on `/api/agent` finally.
4. **Search backend**: simple substring + keyword scoring. No fuzzy / semantic for the first cut.
5. **Storage location**: `chat/index/` — visible (not dot-prefixed), co-located with the data so it survives workspace operations.
6. **Backfill model**: every legacy session is indexed eventually. Work is **distributed** across many small cycles via `CHAT_INDEX_BATCH_SIZE` (default 20). Per-session files are written immediately so partial progress is preserved across restarts and crashes.
7. **History pane fallback**: when an entry has no index yet, fall back to the current first-user-message preview.
8. **MCP tool exposure**: available to **all roles** since it is a passive read-only tool. Each role's `availablePlugins` gets `searchChatHistory` added.
9. **Indexer pause during active runs**: **no pause**. The indexer and `/api/agent` runs share the `claude` CLI but each invocation is independent, so they can run in parallel without contention concerns.
10. **Rate limiting for the MCP search tool**: not needed for the first cut. Each call is local memory only. Revisit if abuse appears.

## Out of scope

- Vector / semantic search
- Automatic suggestion ("you might be looking for…") in the UI sidebar
- Cross-workspace history (only `~/mulmoclaude/chat/`)
- Editing or hand-curating index entries
- Real-time indexing (per-message instead of per-session)
- Multi-user / shared history
- Export / import of the index

## Risks

- **Cost**: each session = one Claude call. A workspace with hundreds of long sessions could rack up nontrivial cost on first run. Mitigated by `limit: 50` per cycle and Haiku-class pricing.
- **Privacy**: session contents are sent to Claude for summarization. Same threat model as the existing chat — users already trust Claude with the chat content. No additional exposure.
- **CLI startup overhead**: `claude` CLI takes ~1-2s to start per call. 50 sessions = ~1 minute of indexing per cycle. Acceptable for a background job, but worth reconsidering if we ever want < 10 minute end-to-end.
- **Stale index for in-progress conversations**: by design — the index reflects the state at the last sync. The frontend can display "indexed N hours ago" if we want to be transparent.
